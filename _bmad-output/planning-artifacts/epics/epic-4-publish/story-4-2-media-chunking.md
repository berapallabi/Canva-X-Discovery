# Story 4.2: Media Chunking Sequence (Vercel Constraints)

### 1. Core User Story Statement
**As a** Canva Creator publishing a large animated GIF or MP4 Video,
**I want the** app to automatically slice and upload my large file seamlessly in the background,
**So that** my post doesn't fail due to standard serverless payload timeout limits on Next.js.

### 2. Acceptance Criteria

*   **AC1: Vercel Physical Limits Evasion (Happy Path)**
    *   **Given** I have a 50MB MP4 selected
    *   **When** I hit publish and pass Pre-flight validation (Story 4.1)
    *   **Then** the frontend MUST manually slice the `ArrayBuffer` generated by Canva into exact `3.9MB` byte chunks.
    *   **And** it MUST send them sequentially to the Next.js Proxy, completely bypassing Vercel's strict 4.5MB payload limitations per request.

*   **AC2: X API INIT / APPEND / FINALIZE Loop (Integration Behavior)**
    *   **Given** the chunking loop is engaged
    *   **When** transmitting the slices
    *   **Then** the Proxy MUST execute X API v1.1 commands sequentially: `INIT` (registering the raw file size), `APPEND` (adding each slice via multiple recursive calls), and `FINALIZE` (closing the media buffer).
    *   **And** the Next.js proxy MUST safely pipe the resulting Media IDs back to the React frontend.

*   **AC3: Proxy Timeout Resetting (UI Behavior)**
    *   **Given** Vercel Serverless limits execution time to 60 seconds total per request
    *   **When** pushing pieces of a massive 400MB video
    *   **Then** the frontend MUST send each slice as a discrete, separate REST call.
    *   **And** this guarantees that each individual query refreshes the serverless execution limit accurately, preventing the Next.js process from timing out and terminating the upload mid-flight.

### 3. Business Context / Objective
Vercel Pro drops dead at 4.5MB payloads. We cannot process standard X maximum files (512MB) unless we explicitly slice and orchestrate the payload transfer natively on the frontend. If we don't build this, video uploads simply won't work.

### 4. In scope
- Client-side blob slicing operations.
- Backend routing to X API v1.1 legacy media upload protocols (which support chunks, unlike X API v2).

### 5. Out of Scope
- Direct S3 bucket logic (we are strictly avoiding custom external storage or databases).

### 6. UI/UX References
- The "Publish" button state must reflect a clear "Uploading (XX%)" loader so the user understands why the process takes a long time.

### 7. Dependencies
- Dependent on Story 4.1 validating the file type appropriately before starting.

### 8. Technical Notes
Use explicit JS `Blob.slice()` in a `while` array loop on the frontend. Send the chunks to `POST /api/publish/append`.

### 9. Testing scenarios
- **Test:** Upload a 30MB sample MP4 file. **Expectation:** Network tab confirms approximately 8 to 10 distinct `POST` chunk requests passing exactly 3.9MB slices to the backend.

### 10. Test Data
- Test video file sized exactly at 60MB to force a long-running execution loop.
